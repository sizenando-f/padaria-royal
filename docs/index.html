Introdução

Padaria Royal

Sistema web completo para gestão de produção e controle de qualidade de fornadas. Registra os parâmetros técnicos de cada produção, coleta avaliações de qualidade e utiliza esse histórico para alimentar um mecanismo de sugestão de dosagem de fermento baseado em similaridade climática ponderada.

Backend
API REST em NestJS com PostgreSQL via Prisma ORM. Autenticação por JWT, validação global por DTO e tarefas agendadas com @nestjs/schedule.
Frontend
Interface web em Next.js 16 com App Router, React 19 e Tailwind CSS 4. Gráficos renderizados com Recharts. Sessão gerenciada via React Context.
Banco de Dados
PostgreSQL 15 em contêiner Docker. Schema versionado com migrations do Prisma. Cinco entidades principais com relações definidas via chave estrangeira.
API Meteorológica
Integração com Open-Meteo para preenchimento automático de temperatura ambiente ao iniciar uma fornada. Gratuita, sem chave de API.

Lógica Central

Algoritmo de Sugestão de Fermento

O módulo de sugestão responde a uma pergunta prática: dado o clima previsto e o tempo de fermentação desejado, qual deve ser a dosagem de fermento por quilo de farinha? A abordagem é baseada em similaridade sobre o histórico de fornadas bem-sucedidas, sem necessidade de treinamento offline.

Etapa 1 — Seleção da base histórica

Apenas fornadas que atendam os três critérios abaixo são consideradas:

  • Nota da avaliação ≥ 4 (Bom ou Excelente)
  • Campo tempAmbienteInicial preenchido
  • Campo tempAmbienteFinal preenchido

Isso garante que o modelo aprenda exclusivamente com produções validadas e com contexto climático completo.

Etapa 2 — Cálculo de distância

Para cada fornada histórica, calcula-se uma distância em relação ao cenário atual. A métrica combina diferença de temperatura e diferença de tempo de fermentação:

distancia = |tempInihist − tempIniatual|
          + |tempFimhist − tempFimatual|
          + |tempohist − tempoalvo| ÷ 60

// A divisão por 60 normaliza minutos para a mesma escala // que os graus Celsius, evitando dominância da variável temporal.

Os registros históricos são então ordenados pela distância e os 20 mais próximos são selecionados para o cálculo.

Etapa 3 — Peso inversamente proporcional

Cada fornada selecionada recebe um peso que aumenta quanto menor for sua distância ao cenário atual:

peso = 1 ÷ (distancia + 0.1)

// O +0.1 evita divisão por zero quando a distância é exatamente 0 // (temperatura e tempo idênticos ao alvo). Nesse caso, peso ≈ 10.

Etapa 4 — Média ponderada da proporção fermento/farinha

O sistema não trabalha com gramas absolutas, mas com a proporção entre fermento e farinha de cada fornada. Isso torna a sugestão independente do volume de produção.

proporcaoi = fermentoi ÷ farinhai

proporcao_media = Σ(proporcaoi × pesoi) ÷ Σ(pesoi)

fermento_sugerido = round(proporcao_media × farinha_atual)

Fallback

SituaçãoComportamento
Nenhuma temperatura informadaTodos os 20 últimos registros recebem peso igual (média simples)
Nenhum histórico disponívelRetorna fermento: 0 e mensagem explicativa
Open-Meteo indisponívelCampos de temperatura ficam null; usuário preenche manualmente
Transparência algorítmica
A resposta sempre inclui as 5 fornadas mais próximas usadas no cálculo (campo provas), com todos os seus parâmetros. Isso permite que o gerente ou padeiro inspecione a origem da recomendação e julgue sua pertinência.
Lógica do Sistema

Regras de Negócio

Janela de edição de 3 dias

Padeiros podem editar produções e avaliações somente se a produção vinculada tem menos de 3 dias a partir de horaInicio. O cálculo é feito em milissegundos:

const diferencaMs = Date.now() - new Date(producao.horaInicio).getTime();
const diferencaDias = diferencaMs / (1000 * 60 * 60 * 24);
if (diferencaDias > 3) throw UnauthorizedException();

Gerentes não têm essa restrição.

Restrição de horário de login

Ao fazer login, o backend compara o horário atual (no fuso do Brasil) com os campos horarioEntrada e horarioSaida do usuário. A comparação é feita entre strings no formato HH:MM, o que é funcionalmente correto para verificar se o horário atual está dentro do intervalo permitido.

Unicidade da avaliação

Antes de criar uma avaliação, o service verifica se já existe um registro com o mesmo producaoId. Como o campo possui constraint @unique no schema, a proteção existe tanto na camada de aplicação quanto no banco de dados.

Autoexclusão bloqueada

Um gerente não pode excluir sua própria conta via DELETE /usuario/:id. O controller compara o id do parâmetro com req.user.sub (ID extraído do token JWT).

Log de auditoria em exclusões

Toda chamada a DELETE /producao/:id cria um registro em LogExclusao antes de executar a deleção, preservando o histórico mesmo que o registro de produção deixe de existir.

Sistema de Permissões

Cada usuário possui 5 permissões booleanas independentes armazenadas no banco. Ao fazer login, essas permissões são emitidas no payload do JWT e também devolvidas no corpo da resposta para uso imediato pelo frontend.

Fluxos

Ciclo de Vida de uma Produção

1
Registro da fornada
O padeiro preenche o formulário com ingredientes e horário. O frontend consulta GET /producao/clima-previsao para preencher as temperaturas automaticamente. O padeiro pode ajustar os valores antes de salvar. POST /producao calcula tempoFermentacaoMinutos e persiste o registro.
2
Estado pendente
A fornada existe no banco sem avaliação. GET /producao/pendentes a retorna. O dashboard exibe o contador de pendências e alerta visual.
3
Avaliação
O operador atribui uma nota de 1 a 5, com comentário e temperatura final real opcional. POST /avaliacao verifica unicidade e persiste. A fornada sai da lista de pendentes.
4
Contribuição ao modelo
Se a nota for ≥ 4 e os campos de temperatura estiverem preenchidos, a fornada passa a compor a base do algoritmo de sugestão em todas as chamadas futuras a GET /producao/sugestao.

Fluxo de Sugestão Inteligente

1
Frontend busca previsão climática
Ao informar os horários de início e fim, o frontend chama GET /producao/clima-previsao. As temperaturas são preenchidas automaticamente nos campos do formulário.
2
Requisição de sugestão
Com farinha, temperaturas e tempo de fermentação preenchidos, o frontend chama GET /producao/sugestao?farinha=20&temp=24&tempFim=26&minutos=180.
3
Cálculo no backend
O algoritmo seleciona os 20 registros mais similares, calcula os pesos, computa a média ponderada e retorna a dosagem sugerida em gramas junto com as 5 fornadas mais próximas como evidência.
4
Exibição e decisão
O padeiro vê a sugestão e pode aceitar, ajustar ou ignorar. O valor pode ser editado livremente antes de submeter o formulário.

Backup Automático

O BackupModule usa @nestjs/schedule para executar uma tarefa agendada diariamente. O processo é:

1
Leitura do destinatário
O service busca o e-mail configurado na tabela Configuracao pela chave Email_Backup.
2
Geração do CSV
Chama internamente a mesma lógica de GET /producao/exportar para gerar o arquivo com todos os dados.
3
Envio por e-mail
O Nodemailer envia o CSV como anexo usando as credenciais configuradas nas variáveis de ambiente (EMAIL_USER e EMAIL_PASS).

O e-mail destinatário pode ser alterado a qualquer momento pelo gerente via PATCH /admin/config/email.

A sessão é controlada pelo AuthContext, um React Context disponível em toda a aplicação via AuthProvider no layout.tsx.

Persistência

Ao fazer login, o token JWT é armazenado em localStorage com a chave royal_token. Os dados básicos do usuário (nome, cargo, permissões) são armazenados em royal_user como JSON. Na inicialização do app, o AuthProvider verifica a presença dessas chaves e restaura a sessão automaticamente.

Proteção de rotas

O useEffect do AuthProvider redireciona para /login caso nenhum token seja encontrado no localStorage. Rotas específicas (como /usuarios e /painel) verificam adicionalmente o cargo do usuário via isGerente.

Expiração do token

Quando o backend retorna 401 em qualquer requisição, o frontend executa logout(), que limpa o localStorage e redireciona para /login.

Operações

Configuração do Ambiente Local

Pré-requisitos
Node.js 20 ou superior, Docker e Docker Compose instalados na máquina.

1. Banco de dados

# Na raiz do projeto
docker-compose up -d

2. Backend

cd backend

# Instalar dependências
npm install

# Criar o .env a partir do exemplo e preencher as variáveis
cp .env.example .env

# Aplicar as migrations no banco
npx prisma migrate deploy

# Popular com dados iniciais (seed)
npx prisma db seed

# Iniciar em modo de desenvolvimento
npm run start:dev

A API estará disponível em http://localhost:3000.

3. Frontend

cd frontend
npm install
npm run dev

A interface estará disponível em http://localhost:3001 (ou 3000 caso o backend esteja em outra porta).

Variáveis de Ambiente

Todas as variáveis são carregadas pelo ConfigModule.forRoot({ isGlobal: true }) do NestJS e ficam disponíveis em todo o backend via ConfigService ou process.env.

VariávelObrigatóriaDescrição
DATABASE_URLSimString de conexão PostgreSQL. Ex: postgresql://admin:adminpassword@localhost:5432/padaria_royal
JWT_SECRETSimChave secreta para assinar e verificar tokens JWT. Use uma string longa e aleatória em produção.
CIDADE_LATSimLatitude da cidade da padaria. Usada na consulta à API Open-Meteo.
CIDADE_LONSimLongitude da cidade da padaria.
EMAIL_USERSimEndereço de e-mail usado pelo Nodemailer para enviar o backup diário.
EMAIL_PASSSimSenha ou token de aplicativo da conta de e-mail.
